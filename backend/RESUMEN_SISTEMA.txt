═══════════════════════════════════════════════════════════════════════════════
 📋 DESCRIPCIÓN DEL SISTEMA
═══════════════════════════════════════════════════════════════════════════════

Sistema de Machine Learning para predecir el consumo óptimo de químicos en
plantas de tratamiento de agua potable, basado en parámetros operacionales.

Predice el consumo diario de 4 químicos:
  • Sulfato de Aluminio (coagulante)
  • Cal (ajuste de pH)
  • Hipoclorito de Calcio (desinfección)
  • Cloro Gas (desinfección)

 🏗️ ARQUITECTURA
═══════════════════════════════════════════════════════════════════════════════

Clean Architecture con 7 capas independientes:

┌─────────────────────────────────────────────────────────────────────────────┐
│  1. CONFIG LAYER (ml/config/)                                               │
│     └─ ml_config.yaml: Configuración centralizada YAML                     │
│                                                                             │
│  2. DOMAIN LAYER (ml/domain/)                                              │
│     └─ entities.py: Entidades de dominio (OperationalData,                │
│        ChemicalConsumption, PredictionResult, AnomalyResult)              │
│                                                                             │
│  3. DATA LAYER (ml/data/)                                                  │
│     ├─ repository.py: PlantDataRepository (extracción de datos)           │
│     └─ preprocessor.py: DataPreprocessor (limpieza, escalado)             │
│                                                                             │
│  4. FEATURES LAYER (ml/features/)                                          │
│     └─ feature_engineer.py: Creación de features (ratios, deltas,         │
│        interacciones, temporales)                                          │
│                                                                             │
│  5. MODELS LAYER (ml/models/)                                              │
│     ├─ trainer.py: ChemicalConsumptionTrainer (3 algoritmos)              │

│     └─ model_manager.py: Versionado y persistencia                        │
│                                                                             │
│  6. INFERENCE LAYER (ml/inference/)                                        │
│     ├─ predictor_service.py: ChemicalConsumptionPredictor (Singleton)     │
│     └─ anomaly_service.py: AnomalyDetectorService (Isolation Forest)      │
│                                                                             │
│  7. UTILS LAYER (ml/utils/)                                                │
│     ├─ config_manager.py: MLConfig Singleton                               │
│     ├─ logger.py: Sistema de logging estructurado                          │
│     └─ validation.py: Validaciones y excepciones personalizadas           │
│                                                                             │
│  8. API LAYER (routers/)                                                   │
│     └─ ml.py: 6 endpoints REST FastAPI                                     │
└─────────────────────────────────────────────────────────────────────────────┘

 📊 DATOS Y FEATURES
═══════════════════════════════════════════════════════════════════════════════

VARIABLES DE ENTRADA (13):
  ├─ Turbidez: turbedad_at, turbedad_ac
  ├─ pH: ph_at, ph_ac
  ├─ Conductividad: conductividad_at, conductividad_ac
  ├─ Temperatura: temperatura
  ├─ Cloro: cloro_residual
  ├─ Presión: presion_entrada, presion_salida
  ├─ Operación: caudal, solidos_totales
  └─ Temporal: mes (para variación estacional)

FEATURES INGENIERÍA (20+):
  ├─ Ratios: turbedad_ratio, conductividad_ratio
  ├─ Deltas: ph_delta, temperatura_delta
  ├─ Interacciones: turbedad_x_sulfato, ph_x_cal, carga_solidos
  ├─ Temporales: mes_sin, mes_cos, dia_semana
  └─ (Opcionales) Rolling windows, lags

VARIABLES OBJETIVO (4):
  ├─ sulfato_aluminio (kg/día)
  ├─ cal (kg/día)
  ├─ hipoclorito_calcio (kg/día)
  └─ cloro_gas (kg/día)


═══════════════════════════════════════════════════════════════════════════════
 🤖 MODELOS ML
═══════════════════════════════════════════════════════════════════════════════

ALGORITMOS IMPLEMENTADOS:

1. Random Forest Regressor
   └─ 100 árboles, max_depth=15, validación por bootstrap
   └─ Feature importance nativo
   └─ Robusto ante outliers

2. XGBoost Regressor
   └─ 100 estimadores, learning_rate=0.1, max_depth=6
   └─ Optimizado para precisión
   └─ Regularización L1/L2

3. LightGBM Regressor
   └─ 100 estimadores, num_leaves=31
   └─ Rápido, eficiente en memoria
   └─ Manejo de missing values

SELECCIÓN AUTOMÁTICA:
  • Se entrenan los 3 modelos
  • Se comparan por RMSE en conjunto de validación
  • Se selecciona el mejor automáticamente
  • Se guarda con metadata y versión


═══════════════════════════════════════════════════════════════════════════════
 📈 MÉTRICAS DE EVALUACIÓN
═══════════════════════════════════════════════════════════════════════════════

MÉTRICAS PRINCIPALES:
  • R² Score: Coeficiente de determinación (0-1, ideal > 0.80)
  • RMSE: Error cuadrático medio en kg (penaliza errores grandes)
  • MAE: Error absoluto medio en kg (error promedio)
  • MAPE: Error porcentual absoluto medio (ideal < 10%)

UMBRALES DE ACEPTACIÓN:
  • R² Score mínimo: 0.70
  • MAPE máximo: 15%

VALIDACIÓN:
  • Train/Val/Test split: 65%/15%/20%
  • Cross-validation 5-fold opcional
  • Análisis de residuales

 🔒 PRINCIPIOS DE DISEÑO APLICADOS
═══════════════════════════════════════════════════════════════════════════════

SOLID PRINCIPLES:
  ✓ Single Responsibility: Cada clase tiene una responsabilidad única
  ✓ Open/Closed: Extensible sin modificar código existente
  ✓ Liskov Substitution: Interfaces bien definidas
  ✓ Interface Segregation: Dependencias mínimas
  ✓ Dependency Inversion: Abstracciones en lugar de implementaciones

DESIGN PATTERNS:
  ✓ Singleton: MLConfig, servicios de inferencia (evitar reload)
  ✓ Repository: Abstracción de acceso a datos
  ✓ Strategy: Selección dinámica de algoritmos ML
  ✓ Facade: Simplificación de API compleja
  ✓ Template Method: Pipeline de entrenamiento

CLEAN ARCHITECTURE:
  ✓ Separación de capas (domain, data, features, models, inference)
  ✓ Dependencias hacia adentro (dominio independiente)
  ✓ Reglas de negocio encapsuladas
  ✓ Infraestructura intercambiable

BEST PRACTICES:
  ✓ Type hints en todo el código
  ✓ Docstrings completos
  ✓ Manejo robusto de errores
  ✓ Logging estructurado
  ✓ Validaciones exhaustivas
  ✓ Configuración externalizada
  ✓ Versionado de modelos
  ✓ Testing preparado
